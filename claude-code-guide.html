Core Tools

File Operations

📖 Read

Read files from the filesystem. Perfect for understanding existing code before making changes.

Use when: You need to examine code, check configurations, or understand file contents before editing.

Read(file_path)

✏️ Edit

Make precise edits to existing files using exact string replacement.

Use when: You need to modify specific parts of a file.

Edit(file_path, old_string, new_string)

📝 Write

Create or overwrite files. Always read first if the file exists.

Use when: Creating new files or completely replacing content.

Write(file_path, content)

Powerful regex-based content search across files.

Use when: Searching for specific code, patterns, or keywords in files.

Run multiple glob searches in parallel for efficiency.

Language Server Protocol (LSP)

LSP plugins supercharge Claude Code with real-time code intelligence, type awareness, and project-wide navigation. They bridge the gap between AI and your actual codebase.

Why LSP Matters

Without LSP, Claude Code operates on static file reading and pattern matching. With LSP enabled, it gains real-time code intelligence from your language's compiler:

❌ Without LSP

  • Reads files as text
  • Pattern-based understanding
  • No type checking
  • Manual cross-file navigation
  • Limited refactoring accuracy
  • No workspace awareness

✅ With LSP

  • Type-aware understanding
  • Real-time error detection
  • Go-to-definition across files
  • Find all references instantly
  • Safe, project-wide refactoring
  • Respects tsconfig/project settings

Practical Examples

Example 1: Type-Aware Understanding // Without LSP: Claude sees "user.profile" but doesn't know properties const email = user.profile.email; // With LSP: Claude knows profile = { email: string, name: string, avatar?: string } // Can suggest: "Did you mean user.profile.avatar?" Example 2: Go-to-Definition Across Files // Without LSP: Must search/grep to find UserService definition import { UserService } from './services'; // With LSP: Instantly knows it's in src/services/UserService.ts:42 Example 3: Real-Time Error Detection // Without LSP: Might suggest code with type errors const result: string = getNumber(); // Type error! // With LSP: "Error: Type 'number' is not assignable to type 'string'" Example 4: Safe Refactoring // Without LSP: Rename might miss references or break types function processUser(user: User) { ... } // With LSP: Safe rename across entire project, updates all references and types

How to Install & Activate

Step 1: Install the Plugin

# Install TypeScript LSP plugin in Claude Code claude plugin install typescript-lsp@claude-plugins-official # Or other language servers: claude plugin install pyright-lsp@claude-plugins-official # Python claude plugin install rust-analyzer-lsp@claude-plugins-official # Rust claude plugin install gopls-lsp@claude-plugins-official # Go

Step 2: Install the Language Server Binary

The plugin needs the actual language server installed on your system:

# TypeScript/JavaScript npm install -g typescript-language-server typescript # Python pip install python-lsp-server # Rust rustup component add rust-analyzer # Go go install golang.org/x/tools/gopls@latest

Step 3: Verify Installation

# Check TypeScript LSP is available typescript-language-server --version tsc --version # Now Claude Code will automatically use it for .ts/.tsx/.js/.jsx files!

  • Install for your primary languages - TypeScript, Python, Rust, etc.
  • Keep language servers updated - They improve with each release

  • Use in large codebases - LSP shines when navigating complex projects
  • Verify installation - Test that the language server binary is in PATH

Memory & Preferences

Claude Code can remember your preferences, coding standards, and workflows across sessions using CLAUDE.md files. Think of it as persistent context that Claude always has available.

Why Memory Matters

Instead of repeating instructions every session, store them once and Claude remembers:

  • ✅ Code style preferences (indentation, naming conventions)
  • ✅ Common commands for your project (build, test, deploy)
  • ✅ Architectural patterns and best practices
  • ✅ Team coding standards and workflows
  • ✅ Security requirements and compliance policies
  • ✅ Personal preferences across all projects

Memory Hierarchy (5 Levels)

Enterprise Policy

Organization-wide instructions managed by IT/DevOps.

Location: /etc/claude-code/CLAUDE.md (Linux), C:\Program Files\ClaudeCode\CLAUDE.md (Windows)

Use for: Company coding standards, security policies, compliance requirements

Shared with: All users in organization

Project Memory

Team-shared instructions for the project.

Location: ./CLAUDE.md or ./.claude/CLAUDE.md

Use for: Project architecture, coding standards, common workflows

Shared with: Team members via source control

Project Rules

Modular, topic-specific project instructions.

Location: ./.claude/rules/*.md

Use for: Language-specific guidelines, testing conventions, API standards

Shared with: Team members via source control

User Memory

Personal preferences for all projects.

Location: ~/.claude/CLAUDE.md

Use for: Code styling preferences, personal tooling shortcuts

Shared with: Just you (all projects)

Project Memory (Local)

Personal project-specific preferences (not committed to git).

Location: ./CLAUDE.local.md

Use for: Your sandbox URLs, preferred test data, local environment settings

Shared with: Just you (current project only)

Quick Start: Create Project Memory

Bootstrap a CLAUDE.md file for your project: # In your project root /init # This creates ./.claude/CLAUDE.md with project context
Example CLAUDE.md for a React project: # Project: Customer Dashboard ## Build & Test Commands - Build: `npm run build` - Test: `npm test` - Dev server: `npm run dev -- --port 3000` - Lint: `npm run lint` ## Code Style - Use 2-space indentation - Prefer functional components with hooks - Use TypeScript strict mode - Name components with PascalCase - Use named exports for components ## Architecture - Components in `src/components/` - Business logic in `src/hooks/` - API calls in `src/services/` - Types in `src/types/` ## Important Context - Main API endpoint: https://api.example.com - Authentication uses JWT tokens - All components should be responsive (mobile-first)

Modular Rules with .claude/rules/

For larger projects, organize instructions into focused topic files:

your-project/ ├── .claude/ │ ├── CLAUDE.md # Main project instructions │ └── rules/ │ ├── code-style.md # Code style guidelines │ ├── testing.md # Testing conventions │ ├── security.md # Security requirements │ └── frontend/ │ ├── react.md # React-specific rules │ └── styles.md # Styling conventions

Path-Specific Rules: Apply rules only to specific files using frontmatter:

--- paths: - "src/api/**/*.ts" - "src/services/**/*.ts" --- # API Development Rules - All API endpoints must include input validation - Use the standard error response format - Include OpenAPI documentation comments - Handle errors with proper HTTP status codes

Memory Best Practices

Writing Effective Memory

  • Be specific: "Use 2-space indentation" > "Format code properly"
  • Use structure: Organize with markdown headings and bullet points
  • Include commands: List frequent commands to avoid repeated searches
  • Document architecture: Explain key patterns specific to your project
  • Review periodically: Update as your project evolves
  • Keep rules focused: One topic per file in .claude/rules/

Organizing Memory Files

  • Project-wide: Use ./CLAUDE.md for team standards
  • Personal global: Use ~/.claude/CLAUDE.md for your preferences
  • Personal local: Use ./CLAUDE.local.md for private project settings
  • Modular rules: Use .claude/rules/ for topic-specific guidelines
  • Path-specific: Add frontmatter with paths: for conditional rules

Real-World Examples

Example 1: Personal User Memory (~/.claude/CLAUDE.md) # My Global Preferences ## Code Style - Always use TypeScript strict mode - Prefer const over let - Use early returns to reduce nesting - Add JSDoc comments for public APIs ## Testing - Write tests before implementing (TDD) - Use descriptive test names: "should X when Y" - Prefer integration tests over unit tests ## Git Workflow - Commit messages: type(scope): description - Always rebase before merging - Squash commits for cleaner history Example 2: Team Project Memory (./.claude/CLAUDE.md) # E-Commerce Platform ## Quick Commands - Start dev: `npm run dev` - Run tests: `npm test -- --watch` - Build prod: `npm run build:prod` - Deploy staging: `./scripts/deploy-staging.sh` ## Architecture - Frontend: Next.js 14 with App Router - Backend: Node.js + PostgreSQL - State: React Context + React Query - Styling: Tailwind CSS ## Important URLs - Staging: https://staging.example.com - Production: https://example.com - API Docs: https://api.example.com/docs ## Code Conventions - Components in `src/app/` follow Next.js conventions - Shared components in `src/components/` - Use Server Components by default - Client Components: add "use client" directive Example 3: Testing Rules (./.claude/rules/testing.md) --- paths: - "**/*.test.ts" - "**/*.spec.ts" --- # Testing Standards ## Test Structure - Arrange-Act-Assert pattern - One assertion per test when possible - Use descriptive test names ## Coverage Requirements - Minimum 80% coverage for new code - 100% coverage for utility functions - E2E tests for critical user flows ## Best Practices - Mock external dependencies - Use test fixtures for complex data - Clean up after tests (afterEach)

Slash Commands

Slash commands are built-in CLI features that provide access to tools and information. Type them directly in your prompt or use them as shortcuts.

Session & Navigation

/resume

Open an interactive session picker to resume a previous conversation. Navigate with arrow keys, press P to preview, R to rename, B to filter by current branch. Can also resume by name: /resume auth-refactor.

/fork

Create an independent branch of the current conversation. Useful for exploring alternative approaches without affecting the main session.

/rename <name>

Set a descriptive name for the current session. Name appears on the prompt bar. Best practice: rename early when starting work on a distinct task.

/clear

Clear the current conversation history and start fresh in the same session.

/compact

Manually compact/summarize the conversation to free up context window space. Auto-compaction happens automatically, but this lets you trigger it on demand.

/context

Show context window usage analysis with token counts, tool usage breakdown, and optimization tips. Helps you understand what's consuming your context budget.

Model & Performance

/model

Switch between available Claude models (Opus 4.6, Sonnet 4.6, Haiku 4.5). Shows the currently active model. Use when you need more power or faster responses.

/effort [low|medium|high]

Control how deeply Claude thinks before responding. Symbols: ○ (low) ◐ (medium) ● (high). Use /effort auto to reset to default. Works with Opus 4.6 and Sonnet 4.6 adaptive reasoning.

/fast

Enable fast mode for quicker responses. Useful when you need speed over depth on Opus 4.6.

/cost

Display token usage and pricing for the current session. Track cumulative API costs.

Planning & Tasks

/plan [description]

Enter Plan Mode where Claude analyzes and plans before executing. Optional description: /plan fix the auth bug. In Plan Mode, Claude uses read-only tools and asks clarifying questions before proposing changes. Press Ctrl+G to open the plan in your editor.

/batch <description>

Decompose a large task into 5–30 independent units, spin up isolated git worktrees for each, execute them in parallel, and create pull requests automatically. Ideal for large-scale refactors or feature implementations across many files.

/simplify

Spawn parallel review agents after implementation to analyze changed files from different angles — checking for reuse opportunities, code quality, and efficiency. Run before committing to improve code quality.

/tasks

View all running or completed background tasks, their IDs and status.

/todo / /todos

View and manage the current task list. Shows in-progress and pending items.

Code & Git

/commit

Invoke the Commit skill to create well-structured git commits with intelligent message generation.

/review-pr [number]

Invoke the Review PR skill for comprehensive pull request analysis with specific recommendations.

/security-review

Audit recent git changes for security vulnerabilities, insecure patterns, and compliance issues. Runs a dedicated security analysis agent over your diff.

/copy

Interactive picker for copying code blocks from the conversation. Press w to write directly to a file instead of copying to clipboard (useful over SSH).

/export

Export the current conversation to a file.

Configuration & Utilities

/config

Configure Claude Code settings interactively with search. Replaces the deprecated /output-style. Toggle thinking mode, update preferences, manage tool settings.

/permissions

Manage tool permissions — review allowed/denied tools and permission rules.

/mcp

Manage MCP (Model Context Protocol) servers. Enable/disable servers, reconnect, view status. See also: claude mcp add CLI command.

/memory

View and manage auto-memory entries. Open memory files for editing.

/voice

Enable/disable voice input mode. Uses push-to-talk (spacebar by default). Supports 20+ languages.

/keybindings

Customize keyboard shortcuts for Claude Code. Supports chord sequences and per-context configuration (chat, bash, vim).

/color <color>

Set the prompt bar color for the current session. Useful for visually distinguishing multiple open sessions.

Diagnose system and installation issues. Detects unreachable permission rules, misconfigured MCP servers, and other common problems.

Display session statistics with optional date range filtering. Track usage over time.

Report issues directly to GitHub. Opens the issue tracker with context pre-filled.

/help

Display all available commands and features. Lists commands alphabetically with descriptions.

Hooks

Hooks are shell commands configured in Claude Code settings that execute automatically in response to specific events. They enable powerful automation and custom workflows.

How Hooks Work

Hooks are triggered by Claude Code events and can execute any shell command. They allow you to:

  • Automatically run tests after code changes
  • Execute linters or formatters on modified files
  • Trigger build processes at key moments
  • Send notifications or integrate with external tools
  • Validate changes before commits
  • Run security checks or dependency audits

Available Hook Events

PreToolUse

Runs after Claude creates tool parameters but before the tool executes. Can return allow, deny, ask, updatedInput, or additionalContext to modify behavior.

Use for: Blocking dangerous operations, logging tool usage, injecting additional context, auto-approving known-safe patterns.

PostToolUse

Runs immediately after a tool completes successfully. Receives the tool name and result.

Use for: Auto-formatting after edits, running tests after file changes, logging tool results.

UserPromptSubmit

Triggered when the user submits a prompt, before Claude processes it.

Use for: Pre-processing user input, injecting context, validation, audit logging.

Stop

Runs when the main Claude Code agent finishes responding. Receives last_assistant_message.

Use for: Post-response automation, triggering CI, sending notifications when Claude finishes a task.

Notification

Fires whenever Claude is waiting for permission, idle and ready for input, or completing authentication. Use matcher to filter: permission_prompt, idle_prompt, auth_success, elicitation_dialog.

Use for: Desktop notifications so you know when Claude needs attention or finishes work.

SessionStart / SessionEnd

Run at the start and end of a Claude Code session. SessionEnd timeout is configurable via CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS.

Use for: Environment setup, cleanup, session audit logs, generating summaries.

PostCompact

Fires after the conversation is compacted/summarized to manage context window usage.

Use for: Saving snapshots, logging compaction events.

WorktreeCreate / WorktreeRemove

Triggered when a git worktree is created or removed. For non-git VCS (SVN, Perforce), configure these to provide custom worktree logic.

Use for: Custom worktree setup, monorepo initialization, environment bootstrapping per worktree.

TaskCompleted / SubagentStop

Fires when a background task completes or a subagent stops. SubagentStop receives last_assistant_message.

Use for: Multi-agent coordination, notifying when parallel agents finish.

Configuring Hooks

Hook configuration in ~/.claude/settings.json (or .claude/settings.json for project): { "hooks": { "PostToolUse": [ { "matcher": "Edit|Write", "hooks": [ { "type": "command", "command": "npx prettier --write $CLAUDE_FILE_PATH 2>/dev/null || true" } ] } ], "PreToolUse": [ { "matcher": "Bash", "hooks": [ { "type": "command", "command": "echo \"Running: $CLAUDE_TOOL_INPUT_COMMAND\" >> ~/.claude/audit.log" } ] } ], "Notification": [ { "matcher": "idle_prompt", "hooks": [ { "type": "command", "command": "notify-send 'Claude Code' 'Task complete — ready for input'" } ] } ] } }

HTTP Hooks

Instead of shell commands, hooks can POST JSON to a URL and receive JSON back. Useful for webhook integrations, CI/CD triggers, or enterprise audit systems.

{ "hooks": { "Stop": [ { "matcher": "", "hooks": [ { "type": "http", "url": "https://your-server.com/claude-webhook", "headers": { "Authorization": "Bearer $MY_WEBHOOK_TOKEN", "Content-Type": "application/json" }, "allowedEnvVars": ["MY_WEBHOOK_TOKEN"] } ] } ] } }

HTTP hooks support env var interpolation in headers via allowedEnvVars. The hook receives event data as JSON and can return JSON to influence behavior.

Hook Examples

Pattern 1: Auto-format on edit (PostToolUse) matcher: "Edit|Write|NotebookEdit" command: "npx prettier --write \"$CLAUDE_FILE_PATH\" 2>/dev/null || true" Pattern 2: Block dangerous commands (PreToolUse on Bash) → Return exit code 2 + stderr to block and show message to Claude Pattern 3: Desktop notification when Claude is idle Notification matcher: "idle_prompt" macOS: "osascript -e 'display notification \"Claude is ready\" with title \"Claude Code\"'" Linux: "notify-send 'Claude Code' 'Ready for your next prompt'" Pattern 4: Audit log every tool call PreToolUse matcher: "" (all tools) command: "jq -r '[.tool_name, .tool_input | tostring] | join(\" \")' >> ~/.claude/audit.log" Pattern 5: Run tests after code changes PostToolUse matcher: "Edit|Write" command: "npm test --silent 2>&1 | tail -5 || true"
⚠️ Important: Hooks can block actions if they fail or take too long. If you encounter unexpected blocking behavior, check your hooks configuration. You can disable or adjust hooks in settings if they cause issues.

Hook Return Values (PreToolUse)

PreToolUse hooks can control tool execution by exit code + stdout JSON: Exit 0 + no output → Allow (proceed normally) Exit 0 + {"decision": "allow"} → Explicitly allow Exit 0 + {"decision": "deny", "reason": "..."} → Block with message shown to Claude Exit 0 + {"decision": "ask", "message": "..."} → Prompt user to approve/deny Exit 0 + {"additionalContext": "..."} → Inject extra context before tool runs Exit 2 + stderr text → Block and show stderr as error to Claude

Hook Best Practices

  • Keep hooks fast and focused - avoid long-running operations that block workflows
  • Use conditional logic - check file types or command names before executing
  • Handle errors gracefully - use || true for non-critical operations
  • Test hooks independently - verify they work before deployment
  • Document your hooks clearly - include comments explaining what they do and why
  • Monitor hook performance closely - slow hooks can impact overall efficiency

Efficient Workflows

Workflow 1: Bug Investigation & Fix

  1. Explore: Use Task(subagent=Explore) to understand where the bug might be
  2. Read: Read the relevant files to understand the code
  3. Hypothesize: Form a theory about what's wrong
  4. Search: Use Grep to find similar patterns or related code
  5. Test: Create a minimal reproduction if needed
  6. Fix: Edit the file with targeted changes
  7. Verify: Run tests with Bash

Workflow 2: Implementing New Features

  1. Plan: Use EnterPlanMode to design the approach
  2. Explore: Understand existing patterns in the codebase
  3. Create TODOs: Use TodoWrite to track implementation steps
  4. Implement: Build the feature step by step, marking TODOs complete
  5. Test: Run tests and verify functionality
  6. Review: Read the changes to ensure quality

Workflow 3: Refactoring Legacy Code

  1. Understand: Use Explore agent to map the current structure
  2. Document: Create TODOs for each refactoring step
  3. Test First: Ensure tests pass before changes
  4. Refactor Incrementally: Small changes, test frequently
  5. Verify: Run full test suite after completion

Best Practices for Efficient Work

🚀 Code Changes

  • Always Read first - understand existing code before modifying
  • Use Edit for targeted changes, not Write for partial updates
  • Keep changes focused - don't refactor surrounding code unless asked
  • Avoid over-engineering - simple solutions beat premature abstractions
  • Use TodoWrite early for multi-step tasks to avoid forgetting steps

🔍 Search & Exploration

  • Use Glob for finding files by pattern (fastest)
  • Use Grep for searching content within files
  • Run multiple searches in parallel when looking for different patterns
  • Use Task(Explore) for open-ended questions about the codebase
  • Save context by using Explore agent instead of manual searches

📋 Planning & Organization

  • Use EnterPlanMode before implementing complex features
  • Use TodoWrite for tasks with 3+ steps
  • Mark TODOs as in_progress immediately when starting work
  • Mark TODOs as completed immediately after finishing (don't batch)
  • Use AskUserQuestion early to avoid rework

⚡ Performance & Efficiency

  • Parallelize independent operations (multiple Glob/Grep calls)
  • Use background tasks for long-running operations
  • Prefer specialized tools over Bash (Read vs cat, Edit vs sed)
  • Use Explore agent for complex codebase questions (saves context)
  • Limit file reads - use line offset/limit for large files

Need to find files? → Glob (by pattern) or Explore agent (if unclear what to search) Need to search code content? → Grep (for specific code) or Explore agent (for patterns/understanding) Need to understand the codebase? → Explore agent (for overview/questions) or read specific files with Read Need to modify code? → Read first → Edit (targeted changes) or Write (new files) Need to run commands? → Bash (terminal operations, git, npm, tests, builds) Complex multi-step task? → Task(subagent_type=...) or EnterPlanMode for design Need to track progress? → TodoWrite for 3+ steps Need user input before proceeding? → AskUserQuestion

Common Parallel Operations

# Find multiple file types at once Glob("src/**/*.tsx", path) Glob("src/**/*.ts", path) // in parallel # Search multiple patterns Grep("export function", path1) Grep("export const", path2) // in parallel # Read multiple files Read("file1.ts") Read("file2.ts") Read("file3.ts") // in parallel

Example: Fixing a Bug Efficiently

Step 1: Explore Task(description="Where is the bug?", subagent_type="Explore") Step 2: Read & Understand Read("/path/to/suspicious/file.ts") Step 3: Search for Related Code Grep("similar_pattern", path) Step 4: Fix Edit(file, old_code, new_code) Step 5: Test Bash("npm test")

Example: Implementing a Feature

Step 1: Enter Plan Mode EnterPlanMode() → Explore codebase patterns → Design approach → ExitPlanMode() Step 2: Create TODOs TodoWrite([ {content: "Create component", status: "pending"}, {content: "Add styles", status: "pending"}, {content: "Write tests", status: "pending"} ]) Step 3: Implement with Progress Tracking (Mark as in_progress, complete immediately after each step) Step 4: Verify Bash("npm test && npm run build")

Teaching Points for Your Team

Key Principles

  • Read first, modify second: Never propose changes without reading the code
  • Parallelize: Run independent operations at the same time
  • Be specific: Use targeted searches and edits, not broad replacements
  • Track progress: Use TODOs for visibility into multi-step work
  • Plan complex work: Use EnterPlanMode for architectural decisions
  • Ask for clarity: Use AskUserQuestion instead of guessing
🎯 Pro Tip: The most efficient engineers using Claude Code are those who plan first (EnterPlanMode, TodoWrite), explore thoroughly (Explore agent), and parallelize operations (multiple Glob/Grep calls at once). Time spent planning saves time in execution.

Long-Running Agent Best Practices

Insights from Anthropic's research on building effective harnesses for autonomous, long-running coding agents. These patterns help agents maintain context, avoid common pitfalls, and deliver reliable results over extended sessions.

Two-Agent Architecture

Anthropic's research identifies a two-agent pattern as foundational for complex, long-running tasks:

1. Initializer Agent

  • Sets up the environment on first run
  • Creates init.sh script for dev servers
  • Establishes progress.txt for work history
  • Makes initial git commit showing setup
  • Defines feature list with clear scope

2. Coding Agent

  • Handles subsequent sessions
  • Focuses on incremental work
  • Reads progress files before starting
  • Commits with descriptive messages
  • Ensures clean handoffs between sessions

Environment Scaffolding

Setting Up for Success

  • Create comprehensive feature lists (200+ items for complex apps) in JSON format to prevent premature completion
  • Mark all features as "failing" initially to establish clear scope and prevent early victory declarations
  • Use strongly-worded constraints: "It is unacceptable to remove or edit tests because this could lead to missing or buggy functionality"
  • Prefer JSON over Markdown for feature tracking - it's more resistant to model-induced corruption
  • Create an init.sh script that handles environment setup, dev server startup, and dependency installation

Incremental Progress Pattern

Work on One Feature at a Time

The research strongly emphasizes incremental work over comprehensive solutions:

  1. Select one feature from the priority list - don't attempt multiple features at once
  2. Implement minimally - get it working before optimizing
  3. Test thoroughly - verify the feature works as expected
  4. Commit to git with a descriptive message explaining what was done
  5. Update progress file with a summary of changes and any blockers
  6. Mark feature complete only after verification passes
  7. Move to next feature - repeat the cycle

Session Startup Protocol

Every session should begin with these checks: 1. Check working directory with `pwd` 2. Review git logs: `git log --oneline -10` 3. Read progress files to understand current state 4. Read feature list and select highest-priority incomplete item 5. Run basic functionality verification before new work 6. Verify app isn't in broken state before implementation Example startup sequence: # Where am I? pwd # What's been done? git log --oneline -5 cat progress.txt # What's the current state? npm run test # or equivalent verification # Now select next feature and begin work

Mandatory Testing

Verification is Non-Negotiable

  • Implement browser automation tools (Puppeteer, Playwright) for end-to-end testing
  • Test features as humans would - not just unit tests or CLI validation
  • Never mark complete without running verification
  • Use visual verification where possible - screenshots, browser previews
  • Explicitly prompt for feature verification before marking as complete

Common Failure Modes & Solutions

"One-shotting" entire application

Problem: Agent attempts to build everything at once, leading to incomplete or broken features.

Solution: Structured feature list with incremental marking. Work on one feature at a time.

Premature victory declaration

Problem: Agent claims work is complete when features are missing or broken.

Solution: Feature tracking with explicit "passing/failing" status prevents early closure. Require verification.

Undocumented bugs cascading

Problem: Issues introduced in one session compound in later sessions due to poor handoffs.

Solution: Git commits + progress notes at every session end. Document blockers and known issues.

Runtime configuration confusion

Problem: Agent struggles to set up or restart the development environment.

Solution: Pre-written init.sh script that handles all setup. Document environment requirements clearly.

Applying to Claude Code

How to apply these patterns in your Claude Code sessions: 1. Use TodoWrite as your feature list → Create comprehensive todos at the start → Mark items in_progress/completed as you go → Never mark complete without testing 2. Use CLAUDE.md as your progress file → Document project state and context → Include known issues and blockers → Update with each significant change 3. Commit frequently with descriptive messages → Each feature gets its own commit → Include what was done and why → Makes it easy to revert if needed 4. Verify before marking complete → Run tests after each change → Use browser automation for UI features → Never assume - always check
🔬 Research Insight: Anthropic's research found that agents equipped with these scaffolding patterns dramatically outperformed those without. The key is treating long-running tasks as a series of small, verifiable steps rather than one large operation. This mirrors how experienced engineers work: incrementally, with frequent commits and constant verification.

When Beneficial

When Most Valuable

Common Pitfalls & Solutions

Context Window Exhaustion

Problem: All tool definitions consumed upfront, leaving minimal space for conversation and reasoning.

Solution: Use Tool Search Tool to load only relevant tools on-demand, preserving 95% of context window.

Solution: Programmatic Tool Calling processes results in sandboxed code; only final output enters Claude's context.

Parameter Ambiguity

Problem: JSON schema defines structure but can't express usage conventions (when to include optional fields, format expectations).

Solution: Provide concrete examples showing realistic patterns Claude should learn from.

Git Worktrees

Run multiple Claude sessions simultaneously on the same repo without collisions. Each session gets its own branch and working directory, sharing the same git history and remote connections.

Why Worktrees?

Without worktrees, two Claude sessions working on the same repo would overwrite each other's changes. With worktrees:

  • Work on a feature in one session while Claude fixes a bug in another
  • Run /batch to spawn dozens of parallel agents, each in isolation
  • Test a risky change without touching your main working directory
  • Each worktree has its own files and branch, zero context bleed

Using the --worktree Flag

Start Claude in a new isolated worktree: # Creates .claude/worktrees/feature-auth/ with branch worktree-feature-auth claude --worktree feature-auth # Start another session in a separate worktree simultaneously claude --worktree bugfix-123 # Auto-generate a random name claude --worktree # → Creates something like .claude/worktrees/bright-running-fox/ # Also available as short flag claude -w feature-auth
Worktree lifecycle: # Worktrees created at: <repo>/.claude/worktrees/<name>/ # Branch name: worktree-<name> # Branches from: default remote branch # On exit: # → No changes: worktree and branch removed automatically # → Changes exist: Claude prompts keep or remove # Add to .gitignore to keep main repo clean: .claude/worktrees/

Subagent Worktrees

Configure subagents to each run in their own worktree for parallel work without conflicts:

In .claude/agents/my-agent.md frontmatter: --- name: feature-builder description: Builds isolated features in parallel isolation: worktree model: sonnet --- Your agent system prompt here...

Or just tell Claude: "use worktrees for your agents" — it handles setup automatically. Each subagent gets its own worktree, cleaned up when done.

Manual Worktree Management

# Create worktree with a new branch (outside Claude) git worktree add ../project-feature-a -b feature-a # Create worktree from existing branch git worktree add ../project-bugfix bugfix-123 # Start Claude in the worktree cd ../project-feature-a && claude # List all worktrees git worktree list # Clean up when done git worktree remove ../project-feature-a
🔑 Key Insight: Worktrees + /batch = your force multiplier. Batch decomposes a task into parallel units, each running in its own worktree, each creating its own PR. What would take hours serially completes in minutes.

Customize shell behavior: # Override shell detection (e.g., prefer zsh over detected bash) export CLAUDE_CODE_SHELL=/bin/zsh # Wrap all bash commands with a logger/auditor export CLAUDE_CODE_SHELL_PREFIX="/path/to/logger.sh" # Return to project root after each Bash command export CLAUDE_BASH_MAINTAIN_PROJECT_WORKING_DIR=1 # Increase bash output limit (default: 30000 chars) export BASH_MAX_OUTPUT_LENGTH=100000 # Default timeout for bash commands export BASH_DEFAULT_TIMEOUT_MS=300000 # Path to shell script sourced before each Bash command # Great for persisting virtualenv/conda activation export CLAUDE_ENV_FILE=~/.claude/env_setup.sh # Detect if running inside a Claude-spawned shell if [ "$CLAUDECODE" = "1" ]; then echo "Running inside Claude Code" fi

Security & Enterprise

Harden Claude Code for enterprise use: # Strip credentials from all subprocess environments # Prevents prompt injection from exfiltrating secrets export CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1 # mTLS authentication export CLAUDE_CODE_CLIENT_CERT=/path/to/cert.pem export CLAUDE_CODE_CLIENT_KEY=/path/to/key.pem export CLAUDE_CODE_CLIENT_KEY_PASSPHRASE=secret # Let proxy handle DNS resolution export CLAUDE_CODE_PROXY_RESOLVES_HOSTS=true # Proxy configuration export HTTPS_PROXY=http://proxy.corp.com:8080 export NO_PROXY=localhost,127.0.0.1,.internal.corp # Override streaming idle timeout (default: 90s) export CLAUDE_STREAM_IDLE_TIMEOUT_MS=120000 # Demo mode (hide email/org in UI, skip onboarding) export IS_DEMO=true

MCP & Plugin Tuning

Fine-tune MCP and plugin behavior: # MCP server startup timeout (default varies) export MCP_TIMEOUT=30000 # MCP tool execution timeout export MCP_TOOL_TIMEOUT=60000 # Max tokens in MCP tool responses (default: 25000) export MAX_MCP_OUTPUT_TOKENS=50000 # OAuth client secret for MCP servers export MCP_CLIENT_SECRET=your-secret export MCP_OAUTH_CALLBACK_PORT=8080 # Plugin git operation timeout (default: 120s) export CLAUDE_CODE_PLUGIN_GIT_TIMEOUT_MS=180000 # Pre-populated plugin seed directories (for container images) export CLAUDE_CODE_PLUGIN_SEED_DIR="/opt/plugins:/usr/share/claude-plugins"

Plugins

Plugins are reusable packages that add skills, agents, hooks, MCP servers, and LSP configurations to Claude Code. Install from marketplaces or create your own.

Managing Plugins

CLI commands for plugin management: # Install a plugin from a marketplace claude plugin install code-review # List installed plugins claude plugin list # Enable/disable a plugin claude plugin enable code-review claude plugin disable code-review # Remove a plugin claude plugin remove code-review # View available plugins from configured marketplaces claude plugin search # Update all plugins claude plugin update

Plugin Architecture

manifest.json

Declares plugin name, description, version, and what it provides (skills, agents, hooks, MCP servers, LSP configs).

CLAUDE_PLUGIN_ROOT

Env var pointing to the plugin's install directory. Use to reference bundled scripts and config files.

CLAUDE_PLUGIN_DATA

Persistent data directory that survives plugin updates. Store state and caches here.

manifest.userConfig

Plugins can declare configuration options. Users set values at enable time; stored securely in keychain.

Plugin Marketplaces

Enterprise Plugin Control

  • Official marketplace: Anthropic's curated marketplace is the default source
  • Custom marketplaces: Point to your own git repos with plugin collections
  • Managed allowlists: Use strictKnownMarketplaces and blockedMarketplaces in managed settings
  • Git subdirectory source: Plugins can point to a subdirectory within a larger git repo
  • Seed directories: Pre-populate plugins in container images with CLAUDE_CODE_PLUGIN_SEED_DIR
Desktop Install: The Claude Code desktop app lets you browse and install plugins without using the terminal. Plugins from configured marketplaces (including the official Anthropic marketplace) appear in a visual browser.

Agent Teams (Experimental)

Agent Teams enable peer-to-peer communication between multiple Claude instances working on the same codebase. Teammates coordinate via a mailbox system and shared task lists.

Enable agent teams: # Set the environment variable export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 # Or in settings.json: { "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } } # Display modes (in settings.json): { "teammateMode": "auto" } # auto-detect best display { "teammateMode": "tmux" } # each teammate in a tmux pane { "teammateMode": "in-process" } # all in one process

How Teams Work

Mailbox System

Teammates communicate via a mailbox. Send messages with Shift+Down to navigate between teammates.

Shared Task List

Coordinate work via CLAUDE_CODE_TASK_LIST_ID. Set the same ID across instances to share tasks.

Plan Approval

Require teammates to get plan approval before executing: CLAUDE_CODE_PLAN_MODE_REQUIRED=true (auto-set by the lead).

TeammateIdle / TaskCompleted

Hook events that fire when teammates finish work. Supports continue: false with stopReason.

New Hook Events (2026)

Recent releases added several new lifecycle hook events beyond the original set:

StopFailure

Fires when a turn ends due to an API error (not a normal stop). Use for alerting and retry logic.

CwdChanged

Fires when the working directory changes. Use to activate virtualenvs, switch tool configs, or update environment.

FileChanged

Fires when a watched file changes. Reactive environment management.

ConfigChange

Fires when config files change. Useful for enterprise auditing.

InstructionsLoaded

Fires when CLAUDE.md or rules files are loaded into context.

PostCompact

Fires after compaction completes. Use to re-inject critical context.

WorktreeCreate / WorktreeRemove

Fires on worktree lifecycle events. Custom VCS setup and teardown.

Elicitation / ElicitationResult

MCP elicitation hooks. Intercept and override interactive MCP dialog responses.

TaskCreated

Fires when a task is created via TaskCreate.

New Hook Handler Types

Conditional Hooks

Hooks now support an if field using permission rule syntax. Example: only fire on Bash(git *) commands.

{ "if": "Bash(git *)", "command": "..." }

HTTP Hooks

POST JSON to a URL instead of running shell commands. Enterprise-friendly for webhook integrations.

{ "type": "http", "url": "https://..." }

Agent Hooks

Spawn a subagent with full tool access for deep analysis before approving/blocking actions.

{ "type": "agent", "prompt": "..." }

New CLI Flags (2026)

--bare

Skip hooks, LSP, plugins, skills, MCP auto-discovery for fast startup. Requires ANTHROPIC_API_KEY. Sets CLAUDE_CODE_SIMPLE=1.

--worktree (-w)

Start Claude in an isolated git worktree with sparse checkout support. Each session gets its own copy of the code.

--channels

Enable channel permission relay. Channel servers can forward tool approval prompts to your phone (research preview).

-n, --name <name>

Set a display name for the session at startup.

claude auth login --console

Authenticate via Anthropic Console (API billing) instead of claude.ai subscription.

claude agents

List all configured agents (built-in and custom).
